# الحوكمة المالية — NALP

قواعد تعديل الأرقام والافتراضات المالية في المشروع لضمان صحة العرض وعدم الرجوع إلى فوضى الأرقام المبعثرة أو المبالغ فيها.

---

## 1. مصدر الحقيقة الوحيد للأرقام المعروضة

- **الإجماليات والعوائد المعروضة للمستثمر** (دخل الملاك، متوسط سنوي، تقييم الخروج، دخل حسب المنطقة/النوع) يجب أن تأتي **فقط** من:
  - `computeProjectTotalsFromEngine()` أو
  - `computeProjectMetrics(scenarioId, years)` عند عرض سيناريو.
- **ممنوع** إدخال أرقام مالية استثمارية **ثابتة (hardcoded)** في صفحات الواجهة أو المكونات (مثل إجمالي دخل 8 سنوات، أو متوسط سنوي، أو valuation) — أي عرض لهذه القيم يجب أن يكون ناتجاً عن استدعاء المحرك أو قراءة من مخرجات المحرك.

---

## 2. أين تُعدّل الأرقام

| نوع التعديل | المكان | ملاحظة |
|-------------|--------|--------|
| افتراضات المشروع والمناطق (cap rate، رأس المال المطلوب، إيرادات، مصاريف) | `nalp/src/lib/finance/assumptions.ts` | سجل الافتراضات المركزي؛ المحرك ولوحة المستثمر يقرآن منه عبر `getDefaultAssumptionsBundle()` أو `getScenarioAssumptions()`. |
| ثوابت مشتقة (مثل PARTNERSHIP_YEARS، REQUIRED_CAPITAL في الواجهة) | تُستمد من الافتراضات أو من `financialCanon` الذي بدوره يقرأ من الافتراضات. | لا تُنشئ ثوابت مالية جديدة في مكونات UI. |
| منطق الحساب (شلال، نقطة تعادل، توزيع) | `nalp/src/lib/calculators/` و `nalp/src/lib/finance/` | أي تغيير في المنطق يتطلب تشغيل الاختبارات المالية. |

---

## 3. قواعد التعديل

1. **لا تُضف أرقاماً استثمارية جديدة في الواجهة** — استخدم دائماً المحرك أو الافتراضات المركزية.
2. **عند إضافة افتراض جديد:** أضفه في `assumptions.ts` مع الوحدة والوصف و`confidenceLevel` و`sourceType`، ثم حدّث `docs/investment-assumptions-register.md` إن لزم.
3. **عند تغيير افتراض حساس (منطقة ج أو د):** راجع `docs/zone-c-zone-d-risk-notes.md` وواجهة "افتراضات حساسة" لضمان بقاء التنبيهات منسجمة.
4. **السيناريوهات (downside/base/upside):** تُبنى من نفس المحرك مع تغيير حزمة الافتراضات فقط — لا تنسخ منطق المحرك.

---

## 4. الضوابط الآلية (Safeguards)

- **اختبارات مالية:** تشغيل `npm run test` يشمل `src/lib/finance/financial.test.ts` و`projectTotalsEngine.test.ts` — يجب أن تمر قبل دمج أي تغيير يمس الحسابات. للتركيز على الاختبارات المالية فقط: `npm run test:financial`.
- **CI:** في `.github/workflows/ci.yml` و`cd.yml` خطوة "Run tests (includes financial tests)" تشغّل `npm run test`؛ عدم مرور أي اختبار (بما فيها المالية) يمنع نجاح الـ build.
- **Pre-commit (اختياري):** يمكن إعداد تشغيل `npm run test:financial` قبل كل commit (مثلاً عبر husky + lint-staged) لضمان عدم كسر الحسابات محلياً.
- **مراجعة الكود:** عند مراجعة PR، التحقق من عدم وجود أرقام مالية ثابتة جديدة في مكونات العرض (بحث عن literals كبيرة أو عن استخدام PROJECT_TOTALS أو ZONE_*_RAW.ownerIncome8Years للعرض المباشر).
